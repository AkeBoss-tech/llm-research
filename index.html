GSM8K: Grade School Math Reasoning

GSM8K is a dataset of 8,500 high-quality grade school math word problems created by human problem writers. These problems require multi-step reasoning and test a model's ability to understand, decompose, and solve mathematical word problems.

Why GSM8K?

  • â€¢ Multi-step reasoning: Problems require breaking down into sub-problems and solving sequentially
  • â€¢ Real-world context: Word problems test natural language understanding alongside math
  • â€¢ Verifiable answers: Each problem has a single correct numerical answer
  • â€¢ Standard benchmark: Widely used to evaluate mathematical reasoning in LLMs

Example Problem

Question: Janet has 16 candy bars. She eats 3 of them. How many does she have left?

Solution:

Janet starts with 16 candy bars.

She eats 3, so we subtract: 16 - 3 = 13

NanoChat is a full-stack, minimal, hackable implementation of an LLM from scratch. It's designed to be fully configurable, tweakable, and trained end-to-end on a budget.

The Mission

Most small models fail at math because they memorize answers rather than algorithms.

We are building a "Micro-Reasoning" laboratory to isolate and verify architectural variables that enable true algorithmic reasoning, drawing on research into System 2 thinking and Latent Space models.

Inference-Time Reasoning

Moving beyond static predictions to models that "burn compute" during inference to solve complex problems.

Controlled Baseline

Using NanoChat to strictly control variables and measure the impact of architectural changes without the noise of massive scale.

Rigorous Verification

Employing "Reverse Operation" tests and OOD generalization to prove the model learned logic, not just patterns.

Novel Generalization

Testing on "Operator Inference" and "Modulo-N" datasets to evaluate in-context learning of new rules.

Experimental Architectures

Moving from System 1 (Fast/Shallow) to System 2 (Slow/Deep) Thinking.

Latent Reasoning (COCONUT)

Cutting Edge

Instead of outputting English words for reasoning, the model outputs "blank" reasoning states (vectors) that are fed back into the input.

  • â€¢ Thinking in high-dimensional vector space.
  • â€¢ Bypasses vocabulary projection for intermediate steps.
  • â€¢ Potential for faster, deeper abstract reasoning.

Recursive Layers

Depth > Width

Looping the input through the same layer weights multiple times to increase computational graph depth without adding parameters.

  • â€¢ Crucial for multi-step sequential logic.
  • â€¢ Forces learning of general reasoning functions.
  • â€¢ Keeps model "Nano" sized while increasing IQ.

Tool Use & Pause Tokens

Agency

Training the model to recognize when to stop generation and call an external calculator or "burn compute" on <think> tokens.

  • â€¢ Solves the large-number hallucination problem.
  • â€¢ Separates logic (planning) from computation (execution).
  • â€¢ Allows manipulation of internal state before answering.

Tool Use: Python Execution & Calculator

NanoChat supports tool use through special tokens that allow the model to pause generation, execute Python code, and resume with the results. This solves the "large number hallucination" problem.

How It Works

1. Model generates special token:

<|python_start|>

2. Generation pauses, code is extracted:

15 * 4 + 3

3. Python executes in sandbox:

result = 63

4. Result inserted, generation resumes:

<|output_start|>63<|output_end|>

Tree of Thought (Lookahead Decoding)

Token-level tree search exploring K candidates for D depth before committing

Implemented
How It Works
  1. At each position, get top K most likely next tokens (e.g., K=3)
  2. For each candidate, simulate D tokens ahead greedily (e.g., D=10)
  3. Calculate cumulative log probability for each branch
  4. Pick the branch with highest score
  5. Commit only the first token of the winning branch, then repeat
Implementation Details
  • â€¢ Configurable K (candidates) and D (depth)
  • â€¢ Scores branches by cumulative log probability
  • â€¢ Compatible with nanochat's Engine class
  • â€¢ 10-30x slower than standard generation

Use Case: Particularly useful for math reasoning tasks where early mistakes can cascade and ruin the entire solution. The lookahead allows the model to "think ahead" before committing to a token.

Data Strategy: The Fuel

1. The "Reverse Operation" Test

We don't just generate 3 + 4 = 7. We generate triples: 3 + 4 = 7, 7 - 4 = 3, 7 - 3 = 4. If the model solves addition but fails subtraction, it hasn't learned the relationship.

2. Process Reward Verification

Generating data as (Question, Step 1, Step 2, Answer). We will use a tiny "Judge" model to score each step as valid/invalid, mimicking state-of-the-art Process Supervision.

3. OOD Generalization

Testing for "Grokking" and Length Generalization. Training on 3-digit multiplication and testing on 4-digit. Using RoPE or ALiBi embeddings to enable length extrapolation.

Future Research Directions

Additional ideas and experiments to improve mathematical reasoning capabilities.

Generator-Validator Architecture

One model generates solutions while another model validates them. The validator scores each reasoning step, and the generator learns to produce higher-quality solutions.

  • â€¢ Generator model produces candidate solutions
  • â€¢ Validator model scores each reasoning step
  • â€¢ Feedback loop improves generation quality

Continuous Chain of Thought

Instead of discrete reasoning steps, allow the model to maintain a continuous reasoning stream that can be refined and iterated upon.

  • â€¢ Maintain reasoning state across tokens
  • â€¢ Allow refinement of intermediate steps
  • â€¢ Enable backtracking and correction

Synthetic Dataset Generation

Generate high-quality synthetic math problems with known solution paths to augment training data and improve generalization.

  • â€¢ Programmatically generate math problems
  • â€¢ Ensure diverse problem types and difficulty
  • â€¢ More finetuning with expanded dataset

Reinforcement Learning Fine-Tuning

Use RL to fine-tune the model by rewarding correct solutions and penalizing incorrect ones, potentially with process-level rewards for correct intermediate steps.

  • â€¢ Reward correct final answers
  • â€¢ Process supervision for intermediate steps
  • â€¢ Learn from mistakes through policy gradients

Abacus Embeddings

Specialized embeddings that encode numerical values in a way that preserves mathematical relationships. Numbers are embedded in a structured space where arithmetic operations map to geometric transformations.

  • â€¢ Structured numerical representations
  • â€¢ Arithmetic as geometric operations
  • â€¢ Better handling of large numbers
  • â€¢ Preserves mathematical relationships

Research Roadmap

A step-by-step plan to evaluate and improve mathematical reasoning capabilities.

1

Algorithmic Baselines

2
